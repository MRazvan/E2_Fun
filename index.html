Remapping selection
① GREEDY + ENTROPY
Entropy-guided remapping selection
Not all remappings are equally useful. A degenerate remapping that maps all colors to the same group — e.g. {1,2,3,4} → {1,1,1,1} — collapses every piece to the same type and discriminates nothing. To avoid this, each candidate remapping is scored by the entropy of its type inventory: H = −∑ (ct/N) log₂(ct/N). A high score means pieces are spread across many distinct types; a low score means most pieces land in the same type. At each step, the highest-scoring remapping that increases the number of discriminated pairs is selected.
Maximum entropy is log₂ 18 ≈ 4.17 bits. High entropy is a good signal but not a guarantee — a remapping can spread pieces evenly and still fail to separate a specific colliding pair, so a discrimination check must follow.
O(N) per candidate avoids degenerate splits near-minimal families in practice locally optimal only entropy ≠ discrimination guarantee
② EXHAUSTIVE
Test all 2c binary remappings
Enumerate every possible color partition and test pairs or triples for the discriminating property. Optimal but exponential in the number of non-boundary colors c.
globally optimal 2c cost
③ ALGEBRAIC
Separating codes on the color graph
Treat remappings as binary labellings of the color graph; a discriminating family corresponds to a separating code — minimum size bounded by ⌈log18(N)⌉.
tight lower bound construction non-trivial
④ GENERALISED k-COLOR
More colors per remapping — T(k) = k(k+1)²(k+2)/4 types
The binary framework (k=2, 18 types) is a special case. Increasing k maps non-boundary colors to a larger alphabet {1…k}, expanding the canonical type space and requiring fewer remappings to discriminate all pieces — the minimum family size drops to ⌈logT(k)(N)⌉.

Possible usages
FULL SOLUTION ENUMERATION
Type-level intersection → exact Sol(P)
Solve each remapped puzzle at the type level — store only which canonical type occupies each cell, never the piece indices. Intersect type-solution sets across all remappings; only boards consistent across every remapping survive. Piece-index assignments (∏ nt!) are computed on demand from the surviving type solutions.
Solve Pf₁ → type solutions S₁ ∩ Solve Pf₂ → type solutions S₂ → S₁ ∩ S₂ = Sol(P) → expand indices
Scalability caveat. For large puzzles (e.g. 16×16) the number of type-level solutions can itself be enormous if the remapped puzzle is under-constrained — even storing and intersecting type solutions may be infeasible without additional pruning.
exact result compact type storage deferred index expansion type count unbounded at scale
STATISTICAL GATHERING
Bayesian aggregation → piece probability
Each remapping produces a per-cell type distribution. Treating remappings as independent observations, combine them as P(piecei at cell) ∝ ∏k Pfk(typefk(piecei) | cell). The result is a confidence ranking over piece placements without ever enumerating full solutions.
Best used when the puzzle is too large for exact enumeration but a ranked candidate list is sufficient — e.g. as a heuristic seed for a local search solver.
scales to large puzzles no full solve required approximate only no guarantee of correctness
Algorithmic approaches
ROW GENERATION + DP
Pattern-A / B row enumeration with multiset DP
Classify grid rows into boundary rows (pattern A: two 0-edges per corner piece, one per edge piece) and inner rows (pattern B: no boundary edges). Enumerate all valid type-inventory vectors per row independently, then use dynamic programming over sorted multisets to find all combinations of row vectors whose type counts sum to the puzzle inventory — reducing exponential search to polynomial DP.
polynomial DP inventory constraint prunes heavily only type-level — no adjacency yet row count grows with W
THREADING
Row solving → boundary keys → vertical threading
Each row of the puzzle is solved independently at the type level: all valid sequences of canonical types that fit the row's boundary constraints and consume a valid subset of the inventory are enumerated. For every such row solution, two boundary keys are extracted — the sequence of south-edge type values along the bottom of the row, and the sequence of north-edge type values along the top. These keys describe exactly what a neighbouring row must present to be compatible.
Threading then assembles full-grid solutions row by row: starting from the top boundary row, each subsequent row is selected only if its top key matches the bottom key of the row above. This propagates constraints strictly downward, eliminating incompatible rows immediately rather than after full grid assembly.
Parallelism. Each row solution is independent — threading can be distributed across solvers, one per candidate top-row or row-pair. The total work scales linearly in the number of distributed solvers, making this approach well suited to parallel execution.
rows solved independently key matching prunes early scales linearly with distributed solvers memory ∝ valid row solutions row count grows with W and H
⚠ Rotations not considered in this document
Important caveat for real puzzles
All examples, type classifications, and solution boards throughout this document treat pieces as fixed-orientation. In a real edge-matching puzzle, pieces are free to rotate (0°, 90°, 180°, 270°), and this creates a subtle but critical problem at the intersection step.
Phantom solutions at the intersection
Placing a piece into a canonical type slot requires a specific type-entry rotation of the original piece. Additionally, each board cell demands a specific board-placement rotation of that canonical type to satisfy its adjacency constraints. The physical rotation of the original piece on the board is the composition of both. For a cross-remapping solution to be valid, this composed rotation must be identical across all remappings.
CONCRETE EXAMPLE — piece P₁
remapping canonical type type-entry rotation board placement rotation original piece rotation
f₁ I2 3 2 1
f₂ I5 2 2 0
The intersection accepts the pair (I2, I5) as valid — both types are present. But f₁ requires the original piece in rotation 1 and f₂ requires rotation 0. These are different physical orientations of the same piece. No single placement satisfies both.
For (I2, I5) to be genuinely valid, I5 would need board placement rotation 3 instead — composing to original rotation 1, matching f₁. Only then does a single physical placement satisfy both remappings simultaneously.
The type-level intersection has no visibility into these rotation compositions — it only checks that the type pair is consistent. A pair that looks valid in the intersection may require two different physical orientations of the same piece: a phantom solution.
Eliminating phantoms requires augmenting the intersection with rotation tracking: for each piece-cell assignment, record the composed original-piece rotation, and enforce that it is identical across all remappings. This is omitted throughout this document for clarity of presentation.
