Column glossary
Remapping structure
k
k
Number of non-boundary colors in the remapped puzzle. At k=2 (binary) all original colors collapse to just two; at k=c the remapping is the identity and you solve the original puzzle directly.
T_I(k)
(k^4 + k^2 + 2k) / 4
Distinct canonical interior piece types under a k-color remapping. Derived via Burnside's lemma on C4. Grows as ~k^4/4 and sets the inventory vector dimension the DP must track.
T(k)
k(k+1)^2(k+2) / 4
Total canonical types across all zones: corners (k^2) + edges (k^3) + interior T_I(k). Used to compute the minimum discriminating family size |F|.
Row & solution space
log(canon.rows)
log C(W-2 + T_I(k) - 1, T_I(k) - 1)
Stars-and-bars count of distinct interior row multisets: the number of ways to fill the W-2 interior cells of one row from T_I(k) types. This is the DP row-vector search space per row before inventory pruning.
log(pot.sol)
log(canon.rows) x (H - 2)
Upper bound on type-level solutions: the product of per-row canonical row counts over all H-2 interior rows. A ceiling only -- the DP prunes heavily via inventory constraints -- but it shows how the solution space scales with height.
Remapping family
log(k^c)
c * log(k)
Log of the full candidate remapping pool. There are k^c possible k-color remappings since each of the c original colors independently maps to one of {1,...,k}. At c=20, k=5 this is ~10^14 candidates -- entropy pre-screening is required to select a good subset without evaluating all of them.
|F|
ceil( log_T(k)(N) )
Minimum discriminating family size: the fewest remappings needed so that every pair of distinct pieces maps to different type signatures in at least one remapping. Decreases as k grows because each remapping can distinguish more pieces. Once |F|=1, a single remapping theoretically separates all N pieces.
Slot occupancy
Sing%
T_I(k) * P(Poisson(lambda)=1) / n_I
Fraction of interior pieces landing in singleton type slots (I_j = 1) under a Poisson occupancy model with lambda = n_I / T_I(k). A singleton means exactly one original piece has that type -- its row assignment is fully determined by the DP, leaving nothing to resolve afterward.
Avg/slot
n_I / E[occupied slots]
Average pieces per occupied type slot. Approaches 1 as k grows toward c. When above 1, multiple pieces share a type and must be distinguished after the DP -- this is what drives within-type ambiguity.
DP complexity
DP dim
min(T_I(k), n_I)
Effective dimension of the inventory vector. Grows with k as more type slots become populated, then saturates at n_I once there are more type slots than pieces. Beyond saturation, additional types are empty and contribute no new DP states.
log(SS)
DP_dim * log(avg/slot + 1)
Log of the nominal DP state space -- the product of (I_j + 1) over active slots, approximated uniformly. This is the worst-case number of distinct inventory vectors before pruning. Inventory constraints eliminate the vast majority of these states in practice.
log(Amb)
T_I(k) * E[log(X!)] where X ~ Poisson(n_I / T_I(k))
Log of within-type ambiguity: the number of ways to assign original pieces to board positions given the DP's type-level solution. Computed as the expected log-factorial per slot under Poisson occupancy, summed over all T_I(k) slots. This is the residual search left after the DP -- it drops super-exponentially with k as pieces spread into more slots and most slots become singletons.
